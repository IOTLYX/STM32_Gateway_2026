# 基于 STM32 + FreeRTOS 的工业边缘数据网关原型 (Industrial Edge Data Gateway)

## 1. 项目简介
本项目面向工业现场或车载网络中的高频异步数据接入场景，旨在解决传统微控制器方案在面对**高频突发数据流**时容易出现的丢包、系统阻塞及实时性下降等问题。项目基于 STM32 微控制器与 FreeRTOS 实时操作系统构建，采用事件驱动型数据流转架构，将“数据接收、协议解析、业务执行”进行分层解耦。网关作为边缘协议接入与转发节点，能够稳定接收南向接口的连续字节流，通过状态机完成有效载荷（Payload）提取，并通过北向接口以标准方式进行数据转发或控制执行（如 CAN 2.0B、GPIO 控制，后续可扩展 MQTT）。

## 2. 核心技术栈
* **微控制器：** STM32F407ZGT6
* **操作系统：** FreeRTOS（基于 CMSIS-OS V2 接口规范）
* **核心底层架构：** UART DMA（Circular 循环模式）+ USART IDLE 空闲中断
* **核心软件算法：** Ring Buffer 环形缓冲区 + 状态机协议解析
* **系统并发与同步：** FreeRTOS 任务通知（Task Notification / Thread Flags）、消息队列（Message Queue）
* **北向业务接口：** CAN 2.0B / GPIO 控制 / 调试串口输出
* **开发与编译工具：** STM32CubeMX、Keil MDK-ARM / STM32CubeIDE

## 3. 具体功能模块

### 3.1 南向通信层（数据物理接入）
* **DMA 连续接收：** 摒弃传统 CPU 轮询或单字节 RXNE 中断方式，配置 UART + DMA 循环接收，将串口接收到的连续数据流自动搬运至内存缓冲区，降低高频接收场景下的 CPU 中断负担。
* **变长数据帧截断检测：** 启用 USART IDLE 空闲中断，利用总线空闲事件捕获一段数据接收完成的时机，为上层执行增量数据处理提供触发条件。
* **轻量级中断处理：** 在 ISR（中断服务程序）中仅完成必要的寄存器状态读取、DMA 当前位置计算及任务唤醒，尽量缩短中断执行时间，保障系统实时响应能力。

### 3.2 核心缓存与解析引擎（数据处理层）
* **异步环形缓冲区：** 使用 Ring Buffer 维护连续字节流的软件缓存，结合 DMA 当前传输计数寄存器（`NDTR/CNDTR`）计算新增数据范围，配合读写指针实现流式数据消费。
* **状态机协议解析器：** 由独立解析任务按字节遍历缓冲区数据，基于状态机机制完成帧头识别、长度读取、有效载荷提取、CRC 校验及帧尾确认。
* **异常数据恢复能力：** 可处理粘包、半包、错码、CRC 错误及帧头丢失等通信异常，出现错误后自动丢弃异常片段并重新同步。
* **跨任务数据解耦：** 将解析出的有效 Payload 封装为统一消息结构体，通过 FreeRTOS 消息队列发送至应用层，实现“接收解析”与“业务执行”的分层隔离。

### 3.3 北向业务层（数据转发与执行）
* **业务执行任务：** 建立独立 Worker Task，长期阻塞等待消息队列，降低空转占用并提升系统调度效率。
* **协议重组与外设驱动：** 任务被唤醒后，根据消息中的命令字与参数执行相应业务逻辑，例如组装标准 CAN 报文发送，或调用 HAL 库控制 GPIO、继电器等外设。
* **可扩展业务接口：** 当前支持 CAN / GPIO / 串口调试输出，后续可扩展 MQTT、RS485 或其他上行接口。

## 4. 软件系统架构与数据流向

系统采用事件驱动模型，运行时可划分为**三个处理阶段**：

1. **[硬件接收层] 数据搬运与事件触发：**  
   串口数据到达后由 DMA 硬件自动搬运至接收缓冲区。当总线进入空闲状态时触发 IDLE 中断。ISR 中仅进行必要的寄存器处理、计算新增数据区间，并通过 `osThreadFlagsSet` 或任务通知机制唤醒解析任务，随后快速退出中断。

2. **[协议解析层] 数据消费与协议剥离：**  
   解析任务平时阻塞等待通知。被唤醒后，根据 Ring Buffer 的读写指针差值（`head != tail`）遍历未处理数据，利用状态机完成协议解析、有效载荷提取与异常数据过滤。解析成功后将结果写入消息队列，再次进入阻塞态等待下一次事件。

3. **[业务执行层] 指令分发与外设操作：**  
   业务任务阻塞等待消息队列。一旦有新消息到达，任务立即被唤醒，根据指令类型执行对应的 CAN 发送、GPIO 控制或其他业务逻辑。执行完成后继续阻塞等待后续消息，实现底层接收与上层业务的解耦。

## 5. 攻克的工程痛点（Interview Focus）

* **痛点一：高波特率场景下 CPU 中断负担过高，易导致实时性下降**
  * **传统方案：** 使用 RXNE 单字节中断逐字节接收，CPU 频繁进入和退出 ISR，系统调度与主循环容易受到影响。
  * **本案解决：** 采用 `DMA Circular + IDLE` 架构，由 DMA 持续搬运数据，仅在总线出现空闲间隔时触发一次中断，减少高频中断带来的系统负担。

* **痛点二：连续字节流存在粘包、半包、错码等异常，协议边界难以识别**
  * **传统方案：** 依赖简单轮询或固定长度假设处理数据，面对异常帧时鲁棒性较差，容易出现丢帧或解析错位。
  * **本案解决：** 引入 Ring Buffer 与状态机解析机制，通过帧头、长度、CRC、帧尾逐级校验，实现异常数据过滤、半包等待及错误后自动重新同步。

* **痛点三：业务层耗时操作可能阻塞底层通信链路**
  * **传统方案：** 在接收逻辑中直接执行 CAN 发送、GPIO 控制等操作，容易因等待外设资源或执行耗时逻辑而拖慢底层数据处理。
  * **本案解决：** 使用消息队列解耦“协议解析”与“业务执行”，解析任务在剥离有效载荷后立即脱手，不参与耗时 I/O 操作，从而提升底层接收链路的连续处理能力。
