# 基于 RTOS 的高可靠性工业边缘数据网关 (Industrial Edge Data Gateway)

## 1. 项目简介
本项目旨在解决工业现场或车载网络通信中，微控制器在面对**高频突发数据洪峰**时容易出现的丢包、系统阻塞及实时性降级等痛点。项目基于 STM32 微控制器与 FreeRTOS 实时操作系统构建，设计了一个高内聚、低耦合的事件驱动型数据流转架构。网关作为边缘协议转换节点，能够稳定接收南向接口的异步海量数据，通过状态机剥离有效载荷 (Payload)，并最终通过北向接口以标准协议（如 CAN 2.0B 或 MQTT）进行精准的数据路由与转发。

## 2. 核心技术栈
* **微控制器:** STM32F407ZGT6
* **操作系统:** FreeRTOS (基于 CMSIS-OS V2 接口规范)
* **核心底层架构:** 串口 DMA (Circular 循环模式) + USART_IDLE 空闲中断机制
* **核心软件算法:** 异步环形缓冲区 (Ring Buffer) + 多级状态机 (State Machine) 解析
* **系统并发与同步:** FreeRTOS 任务通知 (Task Notification / Thread Flags)、消息队列 (Message Queue)
* **开发与编译工具:** STM32CubeMX, Keil MDK-ARM / STM32CubeIDE

## 3. 具体功能模块

### 3.1 南向通信层 (数据物理接入)
* **全硬件接收引擎：** 摒弃传统的 CPU 轮询或单字节 RXNE 中断机制。配置 DMA 直接内存访问，在底层将外设数据透明传输至系统内存，实现接收过程的 CPU 零负载。
* **变长数据帧截断检测：** 启用硬件级 UART_IDLE 空闲中断，利用总线电平状态精准捕获数据帧尾隙，实现对任意长度数据包的无缝截断与低延迟响应。

### 3.2 核心路由与解析引擎 (数据处理层)
* **异步环形缓冲区：** 维护大容量的 Ring Buffer，结合 DMA 硬件传输计数寄存器 (`NDTR`/`CNDTR`)，在中断上下文中实现写指针 (`head`) 的原子级计算与跳跃更新。
* **状态机协议解析器：** 独立运行的解析任务，利用状态机机制逐字节校验通信协议（如识别 `0xAA` 帧头与 `0x55` 帧尾），自动过滤通信链路中的错码、乱码与断包。
* **解耦与跨任务路由：** 将解析出的有效 Payload 封装为标准结构体，通过 FreeRTOS 消息队列推送至应用层，实现“数据接收解析”与“业务逻辑执行”的物理隔离。

### 3.3 北向业务层 (数据转发与执行)
* **业务执行任务：** 建立独立的应用层任务 (Worker Task)，该任务阻塞等待消息队列。
* **协议重组与硬件驱动：** 任务被唤醒后，提取队列中的有效指令，调动底层 HAL 库 API 控制外设（如组装标准 CAN 报文发送至车身总线，或驱动 GPIO 控制外部继电器）。

## 4. 软件系统架构与数据流向

系统在运行时被严格划分为**三个生命周期阶段**，采用标准的事件驱动模型设计：

1. **[硬件中断层] 数据透明传输与事件触发：** 串口数据到达时由 DMA 硬件独立搬运。当一帧完整数据传输结束、总线空闲时触发 IDLE 中断。在 ISR（中断服务程序）内，仅执行必要的寄存器读取与指针更新，随后通过轻量级任务通知（`osThreadFlagsSet`）唤醒阻塞态的解析任务，快速退出中断以保证系统的实时响应能力。
2. **[逻辑解析层] 协议剥离与按需唤醒：** 协议解析任务平时处于挂起态 (Blocked)。接收到中断层发出的同步信号后，任务迅速进入运行态，利用指针偏差 (`head != tail`) 遍历缓冲区数据。提取有效指令并写入消息队列后，任务再次主动挂起，让出 CPU 使用权。
3. **[业务控制层] 指令分发与外设操作：** 外设控制任务长期阻塞在接收消息队列上。一旦队列非空，任务即刻被唤醒，根据出队的指令字执行对应的外设时序控制，执行完毕后继续等待后续指令。

## 5. 攻克的工程痛点 (Interview Focus)

* **痛点一：高波特率下的 CPU 负载过高与数据溢出丢包**
  * **传统方案：** 依赖 RXNE 逐字节触发中断，导致 CPU 频繁进行上下文切换，主循环实时性遭到破坏。
  * **本案解决：** 采用 `DMA_Circular + IDLE` 复合架构，将高频的数据接收完全交由总线矩阵与外设处理。仅在帧间隔时产生微秒级中断脉冲，从根本上释放了 CPU 算力。
* **痛点二：环形缓冲区的指针覆写 (Overrun) 与数据断层**
  * **传统方案：** 依赖时间片轮询 (`osDelay`) 检查缓冲区指针，极易在任务调度间隙被高速运行的 DMA 覆盖未处理的旧数据。
  * **本案解决：** 引入 FreeRTOS 任务通知机制。硬件级中断触发后，高优先级的解析任务立即抢占 CPU 进行数据消费，确保读指针 (`tail`) 能够以最高优先级追踪写指针 (`head`)，消除了追尾隐患。
* **痛点三：外设耗时操作导致整个通信链路的阻塞**
  * **本案解决：** 引入消息队列 (Message Queue) 机制。解析任务将有效载荷剥离后立即脱手，不参与诸如“等待 CAN 邮箱就绪”等耗时的 I/O 阻塞操作，从而保障了底层南向接收链路的绝对畅通与高吞吐率。
